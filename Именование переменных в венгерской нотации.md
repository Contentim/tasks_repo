# 📐 Справка: Венгерская нотация в именовании переменных

> [!summary] Кратко
> Стиль именования, при котором перед описательной частью переменной добавляется строчный префикс, указывающий на тип данных, область видимости или логическую роль. Изобретена Чарльзом Симоньи (Microsoft, 1970–80‑е). Актуальна преимущественно в C/C++, WinAPI и легаси‑кодах.

---

## 🔍 Два подхода к венгерской нотации

| Подход | Что кодирует префикс | Пример | Когда использовать |
|--------|----------------------|--------|-------------------|
| **Systems Hungarian** | Физический тип данных (`int`, `char*`, `bool`, `float`) | `iCount`, `szName`, `bActive`, `fPrice` | Legacy‑проекты, C/C++ без строгой IDE, WinAPI |
| **Apps Hungarian** | Логическую роль или единицу измерения, даже если типы совпадают | `xlLeft` (X‑координата левого края), `rwHeader` (номер строки), `cchMax` (количество символов) | Графика, парсинг, низкоуровневые API, где контекст важнее типа |

> 💡 **Примечание**: Microsoft постепенно отказалась от `Systems Hungarian` в пользу семантического нейминга, но `Apps Hungarian` до сих пор встречается в документации по Win32 и DirectX.

## 📊 Таблица стандартных префиксов

| Префикс | Тип / Назначение | Пример | Примечание |
|---------|------------------|--------|------------|
| `b` / `bln` | `bool` | `bIsReady` | Современный стандарт для флагов |
| `f` | `bool` (flag) или `float` | `fEnabled`, `fValue` | В WinAPI `f` = flag; в научном коде `f` = float. Контекст важен! |
| `i` / `n` | `int` (32‑bit) | `iCount`, `nItems` | `n` часто используется для количества |
| `l` / `lng` | `long` (64‑bit) | `lOffset` | Реже используется из‑за переменного размера `long` в C/C++ |
| `d` / `dbl` | `double` | `dPrecision` | `f`/`flt` оставьте для `float` |
| `c` / `ch` | `char` (1 байт) | `cDelimiter` | |
| `s` / `str` | `string` (высокоуровневый) | `sUserName` | В C/C++ предпочтительнее `sz` |
| `sz` | `null‑terminated string` | `szPath` | `s` + `z` (zero‑terminated). Только для C‑строк |
| `p` | Указатель | `pBuffer` | |
| `pp` | Указатель на указатель | `ppOut` | |
| `h` | Handle (HWND, HFILE, HKEY…) | `hLog` | Специфично для WinAPI |
| `dw` | `DWORD` (`unsigned long`) | `dwFlags` | WinAPI‑специфичный |
| `ui` | `unsigned int` | `uiPort` | |
| `u` | `unsigned` (любой) | `uCount` | |
| `arr` / `a` | Массив | `aScores` | |
| `fn` | Указатель на функцию | `fnCallback` | |
| `m_` | Поле класса (member) | `m_pCache` | Часто комбинируется: `m_bActive` |
| `g_` | Глобальная переменная | `g_hInstance` | |
| `s_` | Статическая переменная | `s_dwInitState` | |

> ⚠️ **Важно**: Префикс всегда пишется **строчными** буквами. Описательная часть — `PascalCase` или `camelCase` в зависимости от гайдлайна проекта.

## 🛠 Правила формирования имени

1. **Префикс + Описательная часть**: `bIsConnected`, `szOutputPath`, `m_dwFlags`
2. **Не дублировать тип**: ❌ `iIntegerCount`, ✅ `nCount`
3. **Булевы переменные**: предпочитай предикатную форму  
   ✅ `bStatusActive`, `bIsConnected`, `bHasPermission`  
   ❌ `bActiveStatus` (звучит как существительное, а не вопрос)
4. **Область видимости** обычно ставится **первой**, затем тип: `g_pConfig`, `m_bInitialized`
5. **Единообразие > идеал**: если в репозитории уже принят `nUserCount`, не меняй на `iUserCount` ради «правильности»

## 🌍 Современный контекст & Best Practices

| Язык / Экосистема | Рекомендация |
|-------------------|--------------|
| C / C++ (WinAPI, драйверы) | Допустимо, особенно для указателей (`p`), хендлов (`h`), `DWORD` (`dw`), булевых (`b`) |
| C# / Java / Kotlin / Go / Rust / TS | **Не рекомендуется**. Строгая типизация + IDE‑тултипы делают префиксы избыточными |
| Python / JS / PHP | Избегать. Динамическая типизация + линтеры лучше справляются с контекстом |
| Легаси‑кодобаза | Следуй существующему стилю. Рефакторинг нейминга без изменения поведения — приоритет №1 |

### ✅ Когда венгерская нотация оправдана
- Работа с `void*` или непрозрачными указателями
- Низкоуровневое API, где тип неочевиден из сигнатуры
- Графика / математика, где нужно различать `xl` (пиксели) и `xd` (документные координаты)
- Legacy‑проекты с жёстким code style

### ❌ Когда лучше отказаться
- В языках с выводом типов и автодополнением
- Когда префикс дублирует информацию, уже известную компилятору/IDE
- В новых проектах без строгого требования legacy‑совместимости

## 📝 Примеры: хорошо vs плохо

```c
// ✅ Системный + Apps Hungarian (WinAPI стиль)
BOOL bIsInitialized;
DWORD dwProcessFlags;
LPTSTR szOutputBuffer;
int cchMaxPath;
HWND hParentWindow;

// ✅ Смешанный с областью видимости
static BOOL s_bCached;
class CSession {
    BOOL m_bConnected;
    DWORD m_dwTimeout;
};

// ❌ Избыточно / противоречиво
int iIntegerValue;          // тип дублируется
float fFloatPrice;          // то же самое
BOOL bActiveStatus;         // нарушает предикатную логику
char* pCharString;          // "CharString" уже подразумевает строку
